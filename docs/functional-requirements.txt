Status note: this document is the product requirements backlog and acceptance target for DayTrail. It is broader than the current implementation. The README and release notes describe shipped behavior; do not treat every item below as a completed public claim until it is covered by tests and release notes.

Below is the **complete comprehensive functional requirements document** for DayTrail, focused only on **what the product must do**, not the technical implementation.

This includes the original work command center requirements plus the later refined direction: **state recovery for your own brain**, context-anchored scratchpad, parallel streams, interrupted state snapshots, clipboard stitching, return-to-work markers, ghost agent monitoring, and developer-focused rescue scenarios. Those additions came from the uploaded notes describing DayTrail as a “state recovery” system for high-context multitasking and short-term memory restoration.  The practical developer use cases like rabbit-hole backtracking, stale hypothesis detection, scattered AI chat finding, clipboard overflow, and ghost agent monitoring are also incorporated.

# DayTrail — Complete Functional Requirements

## 1. Product Vision

DayTrail shall be a **local-first AI work command center** that helps users recover, understand, plan, and explain their work.

The product must help users answer:

```text
What was I doing?
Where did I stop?
What did I complete?
What did I miss?
Who do I need to reply to?
What did I promise?
What is at risk?
Which AI tool helped me?
Which AI thread had that solution?
Why did I change this file?
What should I do next?
What should I tell my manager/client/team?
```

The core promise:

```text
Never lose track of what you did, what you promised, what is pending, where you stopped, and what to do next.
```

The product should not be only a time tracker. It should be:

```text
Personal work memory
Task recovery system
Commitment tracker
AI usage ledger
Context restoration engine
Daily/weekly reporting assistant
Planning assistant
Developer safety net
Client/meeting follow-up assistant
```

# 3. Core User Problems

## 3.1 Context loss

Users jump between tasks, AI tools, code, meetings, emails, and urgent requests. They lose track of:

```text
What they were doing
Why they were doing it
What they already tried
Which AI answer was useful
Where they stopped
What should happen next
```

## 3.2 Missed replies

Users open emails or messages, mentally plan to respond, but forget to reply.

## 3.3 Forgotten commitments

Users say or write:

```text
I will share it tomorrow.
I will check and update.
We will confirm by EOD.
I will send the MoM.
Let me verify and get back.
```

Then those commitments disappear into emails, meetings, calls, and notes.

## 3.4 Invisible AI-assisted work

Users use ChatGPT, Claude, Gemini, Cursor, Copilot, Codex, Cline, Aider, and other tools, but cannot later explain:

```text
Where AI helped
What AI produced
Which output came from AI
Which AI conversation had the solution
```

## 3.5 Invisible offline work

Users attend face-to-face meetings, client visits, field visits, phone calls, and discussions away from the laptop. Normal app trackers show idle time, but the user was actually working.

## 3.6 End-of-day reporting difficulty

Users work all day but cannot explain their progress clearly to management, clients, or themselves.

## 3.7 Parallel work streams

A chronological timeline becomes messy when the user switches between 3–5 different work streams throughout the day. Users need separated streams, not just a single time list.

## 5.4 Context-Anchored Workspace Scratchpad

The system shall provide a scratchpad that changes based on current context.

### Contexts may include

```text
Git repository
Git branch
Workspace folder
Website domain
Email thread
Project/client
Document
Meeting
```

### Behavior

When user switches to Project A, the Project A scratchpad appears.

When user switches to Project B, Project B’s scratchpad appears.

When user returns to Project A, previous Project A notes reappear.

This helps prevent scratchpad notes from becoming detached from the actual work context. 

### Scratchpad functions

```text
Add quick bullet
Pin note
Convert note to task
Convert note to commitment
Attach note to session
Attach note to project
Generate summary from notes
```

## 5.5 Interrupted State Snapshot

The system shall detect sharp context switches and capture a lightweight state snapshot.

### Trigger

A snapshot should be considered when:

```text
User is actively coding/debugging/writing
Then switches to unrelated app/domain
And stays away for more than configured threshold
```

### Snapshot may store

```text
Active workspace
Active file
Cursor line/position if available
Last terminal output tail
Git diff summary
Last AI prompt summary
Current browser tab title/domain
Current task/session
Short summary of what user was doing
```

This feature addresses the pain of leaving a debugging/coding loop and later forgetting the exact state. 

### Privacy

The snapshot should be metadata-first and should avoid storing large raw diffs or sensitive content unless explicitly enabled.

## 5.10 Clipboard Context Stitching

The system shall use clipboard activity to connect related work sessions.

### Behavior

If user copies text/code/error from one app and pastes into another app, the system should infer those sessions are related.

Examples:

```text
Terminal error copied → ChatGPT tab
AI-generated code copied → VS Code
Email text copied → document
SQL copied → DB client
```

This helps link sessions even when app/window titles are not enough. The uploaded notes describe clipboard activity as a strong signal for stitching work context. 

### Functional requirements

The system shall track:

```text
Clipboard content hash
Content type
Source app
Target app
Short preview if enabled
Related session
Related stream
Related project
```

### Privacy requirements

Default should be:

```text
Hash + type + size + short safe preview only
```

Full clipboard history should be opt-in.

# 10. Final Functional Requirement Summary

DayTrail shall function as a **personal AI work memory and recovery system** that captures real work signals, groups them into parallel work streams, restores interrupted context, detects missed replies and promises, tracks AI-assisted outputs, records offline/field work, supports planning, and generates clear daily/weekly reports.

The product’s highest-value experience should be:

```text
When I return to any task, DayTrail tells me exactly where I stopped, what I tried, what mattered, which AI thread helped, what is still pending, and what I should do next.
```

That is the core functional requirement that makes the product irreplaceable.
