# 4. Runtime Architecture

## 4.1 Process model

Use two layers:

```text
worktrace-agent
  Rust background process
  Always running
  Captures signals
  Writes to SQLite
  Runs lightweight rules
  Schedules AI jobs

worktrace-ui
  Tauri window
  Opens only when needed
  Reads from local DB
  Renders markdown/views
  Sends commands to agent
```

The UI should be closable without stopping tracking.

## 4.2 Runtime flow

```text
Watchers
  ↓
Normalizer
  ↓
Privacy Redactor
  ↓
Event Coalescer
  ↓
SQLite WAL
  ↓
Deterministic Engines
  ↓
Prompt Router
  ↓
Markdown/UI Rituals
```

## 4.3 Core principle

Rust should handle facts.

LLM should handle interpretation.

```text
Rust/rules:
- collect events
- normalize data
- apply privacy controls
- coalesce events
- link entities
- detect deterministic states
- prepare compact context

LLM:
- summarize
- classify fuzzy meaning
- generate markdown
- suggest next action
- explain work context
```

Do not let the LLM be the only source of truth.

# 5. Performance Requirements

## 5.1 Resource targets

```text
Idle CPU: < 1%
Normal tracking CPU: < 3–5%
Background memory: 50–120 MB target
Maximum normal memory: < 150 MB
UI memory when open: < 200–300 MB
No continuous LLM inference
No audio capture by default
Dashboard open time: < 1 second for today view
Command search response: < 300 ms for local indexed results
```

## 5.2 Hard performance rules

```text
Do not write every second
Do not run AI continuously
Do not store audio by default
Do not render huge timelines directly
Do not load full history into React state
Do not keep multiple WebViews alive unnecessarily
Do not run long analytics on UI thread
```

## 5.3 Event coalescing

Bad:

```text
1 event per second while Chrome is active for 20 minutes
```

Good:

```text
1 coalesced event:
Chrome / ChatGPT / 20 minutes
```

Coalescing key:

```text
source + app + title + domain + workspace_context
```

## 5.4 Write batching

Flush to DB only when:

```text
30 seconds elapsed
OR 50 events collected
OR app is quitting
OR session boundary detected
```

## 5.5 UI loading

The UI shall:

```text
Use paginated queries
Use virtualized lists
Load today only by default
Lazy-load historical details
Render markdown from compact summaries
Use local indexed search
```

# 14. Deterministic Engines

These must run before AI.

## 14.1 Sessionizer

Groups raw events into sessions.

Rules:

```text
time proximity
same context key
same repo/domain/email thread
clipboard link
AI usage proximity
manual note proximity
calendar/meeting overlap
```

## 14.2 Stream engine

Groups sessions into parallel streams.

Rules:

```text
same repo
same branch
same email thread
same project keywords
same AI chat title
same document
same clipboard chain
```

## 14.3 Idle recovery engine

Detects idle blocks and prompts user.

Rules:

```text
idle > threshold
overlaps calendar
meeting app active before idle
field visit active
common lunch time
returns to same context
```

## 14.4 Unclosed loop engine

Detects:

```text
draft not sent
meeting no notes
AI output not used
task started not completed
promise due no output
agent failed no follow-up
state snapshot unresolved
```

## 14.5 Stale hypothesis engine

Compares:

```text
git diff timestamps
browser/AI/terminal context
uncommitted changes
temporary comments
debug statements
```

## 14.6 Ghost agent engine

Monitors tagged commands/processes:

```text
running
completed
failed
stalled
waiting for input

```

```text
macOS: .dmg / .app
Windows: .msi / .exe
Linux: AppImage / deb / rpm
```

## 21.2 Updates

```text
Signed auto-updates
Staged rollout
Rollback support
Release channel: stable/beta/nightly
```

## 21.3 Permissions onboarding

The app must clearly explain permissions:

```text
Accessibility: active window tracking
Browser extension: active tab metadata
Calendar/email: pending replies and meetings
Notifications: reminders and alerts
Clipboard: optional context stitching
Location/mobile: optional field visit detection
```

# 25. Final Technical Decision

Use:

```text
Tauri v2 + Rust agent + React/TypeScript UI + SQLite + FTS5 + optional DuckDB + local-first LLM adapter
```

But enforce this architecture:

```text
Native-feeling desktop shell
Rust-owned background work
React only for UI
SQLite as fact store
LLM only on demand/batch
No audio capture by default
Metadata-first capture
Event coalescing
Markdown-first intelligence panels
One UI codebase across macOS/Windows/Linux
```

This gives you the balance you want:

```text
Fast development
Robust background processing
Low memory/CPU usage
Cross-platform reuse
Open-source contributor friendliness
Native-like desktop experience
```
